# ADR-001: Apache Kafka в качестве брокера сообщений

> Русская версия · [English version](../../ADR-001-kafka.md)

**Статус:** На рассмотрении  
**Дата:** 2026-06-15

## Контекст

Система должна уведомлять участников (оценщика, осмотрщика, клиента) при смене
статуса заявки или завершении этапа. Сервисы должны оставаться независимыми —
Request Service не должен напрямую вызывать Notification Service, так как это
создаёт тесную связанность и перекладывает на него ответственность за логику
уведомлений.

## Решение

Использовать Apache Kafka в качестве event bus. Каждый бэкенд-сервис публикует
доменные события:

| Сервис | События |
|--------|---------|
| Request Service | `request.created`, `request.status_changed` |
| Inspect Service | `inspect.completed` |
| Review Service | `report.ready` |

Notification Service подписывается на все топики и отправляет email-уведомления.

## Последствия

**Положительные:**
- Сервисы полностью независимы — Notification Service можно перезапустить или
  заменить без влияния на обработку заявок
- События долговечны — уведомления не теряются при временной недоступности
  Notification Service
- Новые потребители (аналитика, аудит-лог) могут подписаться без изменения
  существующих сервисов

**Отрицательные:**
- Добавляет операционную сложность: Kafka требует отдельного деплоя и
  мониторинга
- Для текущего объёма событий может быть избыточным — PostgreSQL `LISTEN/NOTIFY`
  справился бы для более простой конфигурации
