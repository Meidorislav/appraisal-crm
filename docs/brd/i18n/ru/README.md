## 1 Основные положения

**Что разрабатываем?** CRM для автоматизации полного цикла оценки объектов: от приёма заявки до отправки отчёта клиенту.

На рисунках 1–3 показаны бизнес-процессы, которые автоматизируем.

> Рисунок 1 – BPMN: Формирование заявки на оценку  
> Рисунок 2 – BPMN: Процесс осмотра  
> Рисунок 3 – BPMN: Процесс оценки

Компания выполняет оценку объектов в три этапа: приём заявки, выезд осмотрщика, формирование отчёта оценщиком. Сейчас взаимодействие с клиентом происходит по телефону и email, данные хранятся разрозненно, координация между оценщиком и осмотрщиком ручная.

**Цель проекта** – создать CRM, которое переводит весь цикл в единую платформу: от подачи заявки клиентом до получения им готового отчёта.

**Ожидаемые результаты:**

- Снижение времени обработки одной заявки
- Устранение потерь заявок при ручном обмене
- Централизованное хранение данных
- Прозрачность статуса для клиента на каждом этапе
- Структурированное хранение данных осмотров и отчётов

## 3 Границы проекта

**Входит в продукт:**

- Личный кабинет клиента (подача заявки, статус, получение отчёта)
- Интерфейс оценщика (обработка заявок, оценка, отправка отчёта)
- Интерфейс осмотрщика (получение задания, загрузка фото и данных осмотра)
- Admin Panel (управление пользователями, заявками, назначение осмотрщиков)
- Хранение фотоматериалов с осмотра
- Формирование и хранение отчётов
- Веб-приложение, адаптированное под любые устройства
- Уведомления участников о смене статуса заявки

**Не входит в продукт:**

- Интеграция с внешними реестрами (ЕГРН, Росреестр)
- Мобильное приложение
- Десктоп приложение
- Автоматический поиск аналогов через внешние источники
- Электронная подпись отчётов
- Оплата услуг компании

## 4 Описание текущего процесса (AS-IS)

### 4.1 Формирование заявки

Клиент звонит или пишет на почту. Оценщик вручную фиксирует данные, проводит допрос клиента по телефону. Данные хранятся не централизованно.

### 4.2 Процесс осмотра

Оценщик созванивается с клиентом для согласования времени. Если клиент может присутствовать – осмотрщик выезжает. Если нет – процесс приостанавливается до созвона. После осмотра фотографии и данные отправляются оценщику лично.

### 4.3 Процесс оценки

Оценщик получает данные объекта, вручную ищет аналоги, проводит оценку, формирует отчёт и отправляет клиенту.

### Что требует уточнения?

- Описание бизнес-процесса «Опрос клиента при формировании заявки»
- Описание бизнес-процесса «Оценка объекта»
- Как хранятся данные об объектах?

### Ключевые боли AS-IS

- Нет единой точки входа для клиента
- Статус заявки непрозрачен
- Координация между всеми участниками процесса ручная
- Нет единой системы хранения данных

## 5 Бизнес требования

### 5.1 Управление заявками

| ID | Требование |
|----|------------|
| BR-001 | Клиент должен иметь возможность подать заявку через веб-интерфейс, указав тип объекта, адрес и контактные данные, без звонка в компанию. |
| BR-002 | При поступлении новой заявки оценщик должен получать уведомление (email). |
| BR-003 | Система должна сохранять все данные, собранные при допросе клиента, с привязкой к заявке. При подтверждении заявки клиенту должен создаться аккаунт на платформе, данные отправляются на email (при условии, что клиент не зарегистрирован ранее). |
| BR-004 | Заявка должна иметь статусную модель: **Новая → В обработке → Осмотр назначен → Осмотр выполнен → Оценка → Отчёт отправлен → Закрыта**. |
| BR-005 | Клиент должен видеть текущий статус своей заявки в личном кабинете без необходимости звонить. |

### 5.2 Процесс осмотра

| ID | Требование |
|----|------------|
| BR-006 | Оценщик должен иметь возможность назначить осмотрщика на заявку через интерфейс системы. |
| BR-007 | Осмотрщик должен получать задание на осмотр с данными объекта и клиента. |
| BR-007a | Клиент должен получать данные о назначенном осмотрщике. |
| BR-008 | Осмотрщик должен иметь возможность загрузить фотографии объекта, которые автоматически привязываются к заявке. |
| BR-009 | Система должна фиксировать дату и результат попытки связи с клиентом для согласования времени осмотра. |

### 5.3 Процесс оценки

| ID | Требование |
|----|------------|
| BR-010 | Оценщик должен иметь доступ ко всем данным объекта и фотоматериалам осмотра в рамках одной заявки. |
| BR-011 | Система должна поддерживать ввод данных об аналогах и расчёт оценки (конкретные формулы – в отдельном Function Spec). |
| BR-012 | Оценщик должен иметь возможность сформировать и отправить отчёт клиенту через систему. |
| BR-013 | Клиент должен получить уведомление о готовом отчёте и иметь возможность скачать его в личном кабинете. |

### 5.4 Управление пользователями

| ID | Требование |
|----|------------|
| BR-014 | Система должна поддерживать ролевую модель: Клиент, Оценщик, Осмотрщик, Администратор. |
| BR-015 | Администратор должен иметь возможность создавать, блокировать и редактировать учётные записи сотрудников/клиентов. |
| BR-016 | Доступ к данным заявки должен быть ограничен ролью: клиент видит только свои заявки, осмотрщик – только назначенные ему. |

### Что требует уточнения?

- **BR-002.** Возможно, ответить на перечень базовых вопросов (подпроцесс «Опрос клиента при формировании заявки»).
- **BR-002.** Можно предложить другие варианты для уведомлений оценщика (помимо email).

## 6 Требования и ограничения

**Требования:**

- Клиенты имеют доступ к интернету и могут пользоваться веб-интерфейсом.
- Сотрудники компании имеют доступ к интернету и могут пользоваться веб-интерфейсом.

**Ограничения:**

Технологический стек зафиксирован:

| Слой | Технология |
|------|------------|
| Backend | Go |
| База данных | PostgreSQL |
| Кэш | Redis |
| Брокер сообщений | Apache Kafka |
| Аутентификация | Keycloak |
| Поиск | TypeSense *(обдумывается)* |
| Frontend | TypeScript, React |
| Хранилище файлов | S3 Yandex Cloud |

- Все данные (заявки, фото с осмотров, отчёты об оценке) хранятся **5 лет**.
- Компания должна предоставить сервер для деплоя и оплатить S3 Yandex Cloud (стоимость оценивается исходя из объёма хранения за 5 лет).
- Весь исходный код приложения остаётся открытым с лицензией **MIT** или **Apache 2.0**.

## 7 Риски

| ID | Риск | Вероятность | Влияние | Митигация |
|----|------|-------------|---------|-----------|
| R-001 | Формулы расчёта оценки не формализованы | Высокая | Критическое | Зафиксировать формулы в Function Spec до начала разработки сервиса оценки |
| R-002 | Клиенты не переходят на веб-интерфейс | Средняя | Высокое | Простой онбординг, уведомления по email со ссылкой на личный кабинет |
| R-003 | Осмотрщик не может удобно загрузить фото с телефона | Средняя | Среднее | Адаптивный интерфейс, обязательное тестирование на мобильных устройствах |
| R-004 | Сотрудники не готовы менять привычный процесс работы | Средняя | Высокое | Вовлечение сотрудников в согласование интерфейсов на этапе проектирования в Figma |
| R-005 | Высокие расходы на S3 из-за объёма фото за 5 лет хранения | Низкая | Среднее | Оценить объём хранения до деплоя и установить лимиты на загрузку фото на заявку |

## 8 Критерии приемки

### Подача заявки
- Клиент подаёт заявку через веб-интерфейс без звонка в компанию.
- Оценщик получает уведомление о новой заявке в течение 1 минуты.

### Управление заявкой
- Оценщик назначает осмотрщика через интерфейс системы.
- Статус заявки меняется автоматически при переходе между этапами.

### Осмотр
- Осмотрщик загружает фотографии объекта через веб-интерфейс с мобильного устройства.
- Фотографии отображаются в карточке заявки у оценщика.

### Оценка
- Оценщик формирует и отправляет отчёт через систему.
- Клиент получает уведомление и может скачать отчёт в личном кабинете.

### Ролевая модель
- Клиент видит только свои заявки.
- Осмотрщик видит только назначенные ему осмотры.
- Оценщик имеет доступ ко всем заявкам.

### Производительность
- Время загрузки любой страницы не превышает 3 секунды при стандартном интернет-соединении.

### Деплой
- Система развёрнута на сервере заказчика, доступна по HTTPS, настроен CI/CD pipeline.

## 9 Расходы компании

Основные расходы на поддержку проекта:

- Оплата сервера и домена (если у компании нет своего)
- Оплата S3 bucket (если у компании нет своего хранилища)

Расходы носят ежемесячный/ежегодный характер. Уточнение цен — после согласования политики хранения данных.
